# Spec-Kit: Plan

Generate an implementation plan from the current branch's specification.

## Pre-Flight

### Current branch
!`~/.claude/hooks/speckit-helper.sh branch`

### Auto-detect spec branch
!`~/.claude/hooks/speckit-helper.sh check-spec`

### Available specs
!`~/.claude/hooks/speckit-helper.sh list-specs`

### Constitution
!`~/.claude/hooks/speckit-helper.sh constitution`

### Start plan phase (RIPER-style write-block)
!`~/.claude/hooks/speckit-helper.sh plan-phase-start`

## Instructions

**Required inputs**: `spec.md` and `constitution.md` must both exist. If either is missing, tell the user which command to run first (`/speckit.specify` or `/speckit.constitution`).

### Phase 0: Research (Truth Map)

The goal of this phase is to compress verifiable truth about the codebase into a single artifact so later phases reason over a small, dense context instead of re-exploring the repo every turn.

1. **Load the spec** from `.specify/specs/<branch>/spec.md`
2. **Build the Truth Map** at `.specify/specs/<branch>/research.md` — **always**, even when no clarifications are needed. The Truth Map is the canonical context for Phase 1 and every downstream command:

   ```markdown
   # Research: <branch-name>
   <!-- Generated by /speckit.plan Phase 0 -->
   <!-- Date: YYYY-MM-DD -->

   ## Affected Modules
   | Path | Role | Key Symbols (file:line) |
   |------|------|-------------------------|
   | `src/auth/session.ts` | Session lifecycle | `createSession:42`, `revokeSession:118` |

   ## Relevant Symbols
   - `path/to/file.ext:LINE` — one-line description of what this symbol does and why it matters here

   ## Existing Patterns & Conventions
   - <pattern observed in the code (error handling, DI, logging), with a file:line citation>

   ## Open Questions
   - <Questions that cannot be answered from the code alone — become `[NEEDS CLARIFICATION]` in the spec>
   ```

3. **Resolve any `[NEEDS CLARIFICATION]` markers** while building the Truth Map:
   - Research the codebase using Grep/Glob/Read to find answers
   - If questions can be answered from code, resolve them inline in research.md
   - If questions require user input, use AskUserQuestion and record the resolution

**Why this matters:** research.md is the input agents should read before Phase 1. After it is written, the user can `/clear` and resume with research.md as the sole source of truth — no re-exploration, no context pollution from intermediate search results. Every claim in the Truth Map MUST carry a `file:line` citation so downstream agents can verify it without re-grepping the repo.

### Phase 1: Design

3. **Analyze affected code**:
   - Identify files that need modification
   - Understand existing patterns and conventions
   - Map functional requirements to code locations

4. **Create design artifacts** (only if the feature warrants them):
   - `data-model.md` — entity relationships, schema changes (skip if no data model changes)
   - `contracts/` directory — API contracts, interface definitions (skip if no API changes)

5. **Write `plan.md`** at `.specify/specs/<branch>/plan.md`:

   ```markdown
   # Plan: <branch-name>
   <!-- Generated by /speckit.plan -->
   <!-- Date: YYYY-MM-DD -->

   ## Research Notes
   <Resolved clarifications, technical findings, key decisions>

   ## Affected Files
   | File | Change Type | Description |
   |------|------------|-------------|
   | `path/to/file` | modify | <what changes> |
   | `path/to/new` | create | <what it does> |

   ## Data Model
   <Schema changes, entity relationships — or "No data model changes required">

   ## API Contracts
   <Endpoint changes, interface updates — or "No API changes required">

   ## Implementation Approach
   <High-level strategy, key decisions, trade-offs>

   ## Code Sketches (Mental Alignment)
   <!--
   REQUIRED for every non-trivial change. A 5-10 line snippet per major change
   demonstrates the architectural intent BEFORE implementation tokens are spent.
   The human architect verifies these sketches; disagreements are resolved here,
   not mid-implementation. Sketches are illustrative — not final code.
   -->

   ### Sketch 1: <change name>
   **File:** `path/to/file.ext` (new | modify at `:LINE`)
   **Intent:** <one sentence: what this snippet demonstrates>

   ```<lang>
   // 5–10 lines showing the shape of the change:
   // - new function signature, class skeleton, or route handler
   // - key call sites into existing code (with file:line references)
   // - error / return contract
   ```

   **Why this shape:** <1-2 lines tying the sketch to the Implementation Approach and the research.md Truth Map>

   ### Sketch 2: <change name>
   …

   > If a change is truly trivial (rename, typo, single-line config), note it here as
   > "No sketch — trivial" with a one-line justification. `/speckit.fix` is the preferred
   > path for trivial changes.

   ## Constitution Compliance
   - [x] **Principle 1**: <how this plan aligns>
   - [x] **Principle 2**: <how this plan aligns>
   - [ ] **Principle N**: <not applicable — explain why>

   ## Risks & Mitigations
   | Risk | Impact | Mitigation |
   |------|--------|------------|
   | <risk> | <impact> | <mitigation> |

   ## Quick Start
   1. <First step to begin implementation>
   2. <Second step>
   ```

6. **Constitution compliance gate**: every principle must be addressed (either aligned or explicitly marked not applicable with justification). If the plan violates a principle, flag it to the user and ask for confirmation before proceeding.

7. **Mental Alignment gate**: every non-trivial change in the Affected Files table must have a corresponding sketch in the Code Sketches section (or an explicit "No sketch — trivial" justification). A plan without sketches is not ready for `/speckit.tasks`.

8. **Present plan summary** to user:
   - Files affected (count + list)
   - Key design decisions
   - Code sketches (count + one-line intent each)
   - Any constitution compliance concerns
   - Suggest next step: `/speckit.tasks`

9. **End plan phase**: after `plan.md` is written and the summary is presented, clear the write-block marker so downstream commands can edit code:

   ```bash
   ~/.claude/hooks/speckit-helper.sh plan-phase-end
   ```

   If you abort the plan before completion, still run the command above — or delete `.specify/.plan-in-progress` manually — otherwise the next Edit/Write in this project will be blocked.
